# i18n — How this codebase handles translations

> Last updated: 2026-04-27 (i18n full-overhaul)

This document is the practical reference for adding, changing, or
auditing translations in betsplug.com. For the historical record
of what the 2026-04-27 overhaul fixed, see `i18n-audit/`.

## TL;DR

- **Library**: custom in `frontend/src/i18n/`. **No** next-intl /
  react-intl / i18next.
- **Source of truth**: `messages.ts` — the EN dict (`const en = {...}
  as const`) defines `TranslationKey = keyof typeof en`. Every other
  locale is `Partial<Record<TranslationKey, string>>`.
- **Enabled locales (6)**: `en, nl, de, fr, es, it` — these get
  hand-translated, must be 100% covered, and are surfaced in the
  language switcher.
- **Parked locales (10)**: `sw, id, pt, tr, pl, ro, ru, el, da, sv` —
  hidden from the language switcher, files kept for future
  re-activation, auto-filled by the translator.
- **Lookup**: `useTranslations()` → `t(key, vars?)` in client
  components; `translate(locale, key)` directly in server code.
- **Loud failures in dev**: missing keys log `console.error` and a
  bogus key prints `[i18n:missing]…[/i18n:missing]` so it stands out.
- **Pre-commit + CI**: blocks new hardcoded JSX strings + missing
  keys for enabled locales.

## Adding a new UI string

1. Open `frontend/src/i18n/messages.ts`.
2. Add to the EN block (around line 13-3244) under a logical
   namespace (`pred.*`, `footer.*`, `a11y.*`, etc.):
   ```ts
   "myFeature.title": "Match preview",
   ```
3. Add to the NL inline block (around 3250+):
   ```ts
   "myFeature.title": "Wedstrijdvoorbeeld",
   ```
4. Add to each of the 4 aux enabled locales (`de.ts`, `fr.ts`,
   `es.ts`, `it.ts`) — hand-translate. The auto-translator will
   fill the 10 parked locales on commit.
5. Use the key in a client component:
   ```tsx
   import { useTranslations } from "@/i18n/locale-provider";
   function MyComp() {
     const { t } = useTranslations();
     return <h1>{t("myFeature.title")}</h1>;
   }
   ```
6. Run `npm run i18n:check` to confirm.

## What NOT to do

- **Don't hardcode user-facing strings** in JSX or attributes:
  ```tsx
  // ❌
  <button aria-label="Next page">…</button>
  <h1>Match preview</h1>

  // ✅
  <button aria-label={t("a11y.nextPage")}>…</button>
  <h1>{t("myFeature.title")}</h1>
  ```
- **Don't use `as any` to bypass `TranslationKey`**:
  ```tsx
  // ❌
  t("some.key" as any)
  // ✅
  t("some.key")  // and ensure the key exists in en.json
  ```
- **Don't add a key to only EN** — pre-commit will block. Either
  add to all 6 enabled locales (hand-translate) or rely on the
  auto-translator (rate-limited, less precise).
- **Don't touch a parked locale's file** without first promoting
  it to enabled (criteria below).
- **Don't translate brand names**: `BetsPlug`, `Pulse`, `BTTS`,
  `ROI`, `Free`, `Silver`, `Gold`, `Platinum`, league names —
  whitelisted in `i18n-check.mjs`.

## Adding a new locale

Three-step process — none of these alone is enough:

1. **Add code-paths**: append to `locales` tuple in
   `src/i18n/config.ts`, add `localeMeta` entry, create
   `src/i18n/locales/<loc>.ts` mirroring an existing aux locale.
2. **Hand-translate the dictionary** — minimum: every key in EN.
   Auto-translator (`npm run translate`) gives a starting point but
   the result needs human review for tone.
3. **Promote to enabled** by adding to `ENABLED_LOCALES` in
   `config.ts` ONLY when:
   - URL slugs are translated in `src/i18n/routes.ts`
   - PAGE_META block hand-translated for every route in
     `src/data/page-meta.ts`
   - All `Priority-1+2` hardcoded strings (audit list in
     `i18n-audit/03-hardcoded-strings.md`) are extracted and
     translated for this locale
   - Sanity content (homepage, league hubs, bet-type hubs)
     hand-translated
   - `npm run i18n:check:full` passes

## Promoting a parked locale to enabled

Pick one parked locale at a time. Use the four criteria above. The
SEO recovery handoff (`seo-audit/09-handoff.md`) suggests this
order: pt → pl → ru → el → ro → tr → da → sv → id → sw.

A scheduled remote agent (`betsplug-seo-pt-prep`, runs 2026-05-24)
will inventory what's still missing for PT specifically.

## Tone-of-voice per locale

There's no separate tone guide; each locale's existing translations
are the reference. Read 5-10 entries in the same namespace before
adding new ones to keep the voice consistent.

## What lives where

| Content type | Bron | i18n method |
|--------------|------|-------------|
| UI strings (buttons, labels, form text, banners) | `messages.ts` + `locales/*.ts` | `t("key")` |
| SEO meta (title, description, og:*) per route | `data/page-meta.ts` | `PAGE_META["/path"][locale]` |
| Editorial content (homepage blocks, league hubs, bet-type hubs) | Sanity CMS | Per-locale localized fields |
| Article body, title, excerpt, category | Sanity CMS — **mono-lingual today** | EN-only per design; multi-lingual schema is a separate sprint |
| Legal pages (privacy, terms, cookies, responsible-gambling) | hardcoded in `app/<page>/page.tsx` | **EN-only by design** (see sitemap.xml comment) |

## How the runtime works

1. URL hits middleware (`src/middleware.ts`).
2. Middleware extracts `/<locale>/...` prefix, rewrites to canonical
   EN path, sets `x-locale` request header + `NEXT_LOCALE` cookie.
   Parked locales additionally get `X-Robots-Tag: noindex, follow`.
3. Server components read locale via `getServerLocale()` (in
   `src/lib/seo-helpers.ts`) — header first, cookie fallback,
   default EN.
4. Root `app/layout.tsx` wraps children in
   `<LocaleProvider locale={locale}>`.
5. Client components call `useTranslations()` → `t(key)` →
   `translate(locale, key)` → `dict[key] ?? en[key]`.
6. In dev, missing keys log a `console.error` and the bogus-key
   case renders `[i18n:missing]key[/i18n:missing]` for visibility.
7. `<LocaleSanityCheck />` (mounted in dev only) reads rendered
   `<main>` text and franc-detects the language; logs an error if
   the dominant detected language differs from the URL locale.

## Adding a new key — common patterns

### Plain text

```tsx
{t("myFeature.title")}
```

### With placeholders

```ts
// messages.ts
"myFeature.greeting": "Hello, {name}!",
```
```tsx
{t("myFeature.greeting", { name: user.firstName })}
```

### In an attribute

```tsx
<input placeholder={t("myFeature.searchPh")} aria-label={t("a11y.searchInput")} />
```

### Pluralisation

The `formatMsg` helper supports placeholder substitution but **not**
ICU plural-rules. For a count-dependent label, define separate keys:

```ts
"matches.zero": "No matches",
"matches.one": "1 match",
"matches.many": "{count} matches",
```
```tsx
const key = count === 0 ? "matches.zero" : count === 1 ? "matches.one" : "matches.many";
{t(key, { count })}
```

If you need true plural-rules across locales, that's a library
upgrade — bring it up before reaching for it.

## Glossary

| Term | Definition |
|------|------------|
| **ENABLED_LOCALES** | The 6 locales we hand-translate and surface in the language switcher (en/nl/de/fr/es/it). Defined in `src/i18n/config.ts`. |
| **Parked locale** | One of 10 locales (sw/id/pt/tr/pl/ro/ru/el/da/sv) whose `.ts` file exists, gets auto-translated, but is hidden from the language switcher. Direct URL hits still work. |
| **Brand passthrough** | A string that legitimately stays verbatim in every locale (BetsPlug, Pulse, Free, Silver, Gold, Premier League, etc.). Whitelisted in `i18n-check.mjs`. |
| **TranslationKey** | TypeScript type derived from `keyof typeof en`. Every `t(key)` call must use a member of this union — no `as any`. |
| **`.i18nignore`** | Baseline whitelist of file:line entries that are exempt from the hardcoded-string scan. Existing tech debt only — DON'T add new entries to silence CI. |

## When to escalate

- Want to migrate to next-intl / react-intl? → ADR + team discussion.
  82 components depend on `useTranslations`; not a one-PR change.
- Want to add a 17th locale? → Cas + Dennis decision; current
  ENABLED_LOCALES set was deliberately scoped at 6.
- Want to make blog post / article content multilingual? → Sanity
  schema migration + ~115 article translations. Currently mono-
  lingual EN by design (see audit fase 8 follow-up note).
